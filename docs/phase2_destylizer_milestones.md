## 全体アーキテクチャ（再掲）

```
音声入力(16kHz)
  │
  ▼
HuBERT-Large-ASR 第18層（凍結）  ← マイルストーン1
  │  出力: [B, T, 768] @ 50Hz
  ▼
Conformer ×6（ALiBi位置エンコーディング）  ← マイルストーン2
  │  出力: [B, T, 768]
  ▼
コンテンツ特徴量 fc  ← ★推論時はここでStylizerへ渡す
  │
  ▼
FSQ 下方射影（768→3）→ 量子化[5,3,3] → 上方射影（3→768）  ← マイルストーン3
  │  STE勾配伝搬
  ▼
ASRデコーダ（4層トランスフォーマー）  ← マイルストーン4
  │
  ▼
テキスト予測（ASR損失で学習）  ← マイルストーン5
```

## マイルストーン1: HuBERT特徴量抽出器の統合

### タスク名
HuBERT-Large-ASR第18層からの凍結特徴量抽出パイプラインの構築

### 具体的な成果物
- `src/destylizer/hubert_extractor.py`: HuBERT-Large-ASRモデルのロードと第18層出力抽出クラス
- `src/destylizer/audio_utils.py`: 16kHzリサンプリング、パディング、バッチ処理のユーティリティ
- `tests/test_hubert_extractor.py`: 単体テスト（出力形状、フレームレート、再現性検証）
- 特徴量キャッシュ機構（大規模学習データの前処理用）

### 受け入れ基準
| 基準 | 目標値 |
|---|---|
| 使用モデル | `facebook/hubert-large-ls960-ft`（HuggingFace） |
| 抽出層 | 第18層（0-indexedで17）、全24層のうち |
| 出力次元 | `[B, T, 768]` |
| フレームレート | 50Hz（20ms/フレーム） |
| モデル状態 | 完全凍結（`requires_grad=False`、`model.eval()`） |
| メモリ効率 | `torch.no_grad()`コンテキスト内で推論、勾配計算なし |
| 数値再現性 | 同一入力に対し同一出力（決定論的推論） |
| バッチ処理 | 可変長音声のパディング・マスク処理が正しく動作 |
| 特徴量キャッシュ | LMG 1,300時間分の特徴量を `.pt` または `.npy` で保存・再利用可能 |

### 推定作業量
**2日**

### 依存関係
- フェーズ1完了（音声データのダウンロード・前処理パイプライン）
- `transformers>=4.36`（HuBERTモデル対応）
- `torchaudio>=2.1`（音声I/O）

### 技術的リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| HuBERT-Large-ASRの「第18層」定義の曖昧さ（CNNエンコーダ層を含むか否か） | 中 | `transformers`の`HubertModel`は`output_hidden_states=True`で全層出力を返す。CNNエンコーダ出力が`hidden_states[0]`、以降がトランスフォーマー各層。第18層は`hidden_states[18]`（トランスフォーマー第18層）と解釈。CosyVoice 2の実装を参考に確認する |
| メモリ不足（HuBERT-Largeは約1.2GB、長い音声でGPUメモリ逼迫） | 中 | 音声を最大30秒にチャンク分割して特徴量抽出。キャッシュ前処理で学習時のGPU負荷を回避 |
| `hubert-large-ls960-ft`がファインチューニング済みモデルで層出力の挙動が異なる可能性 | 低 | ファインチューニング前後でhidden statesの統計量（平均・分散）を比較検証。CTC headは使用しないことを確認 |
| 特徴量キャッシュのディスク容量（1,300時間 × 50Hz × 768次元 × 4byte ≒ 700GB） | 高 | float16でキャッシュ（350GB）、またはオンライン抽出（HuBERTを毎回推論）との選択。GPUメモリに余裕があればオンライン抽出を推奨 |

### ✅ 完了ノート
Phase 1で`stylestream/data/hubert_extractor.py`として実装済み。Phase 2ではDestylizerモデル内でHuBERT特徴量を入力として受け取る設計。`model.py`の`_ensure_time_last()`が(B,768,T)/(B,T,768)の両レイアウトに対応。`input_norm`（LayerNorm）でHuBERT出力を正規化。

## マイルストーン2: Conformerブロック実装（ALiBi付き）

### タスク名
ALiBi位置エンコーディング付きConformerブロック×6の実装

### 具体的な成果物
- `src/destylizer/conformer.py`: Conformerブロック実装
  - マルチヘッド自己注意機構（ALiBi位置バイアス付き）
  - 畳み込みモジュール（深さ方向分離可能畳み込み）
  - フィードフォワードモジュール×2（マカロン構造）
  - LayerNorm、ドロップアウト、残差接続
- `src/destylizer/alibi.py`: ALiBi位置エンコーディング実装
- `src/destylizer/conformer_encoder.py`: 6層Conformerスタック
- `tests/test_conformer.py`: 単体テスト（出力形状、ALiBiバイアス正当性、勾配フロー）

### 受け入れ基準
| 基準 | 目標値 |
|---|---|
| Conformer層数 | 6 |
| 隠れ層次元 | 768 |
| FFN次元 | 3072（拡大比4倍） |
| 注意ヘッド数 | 12（768/64、ヘッド次元64と推定） |
| 畳み込みカーネルサイズ | 31（Conformer原論文デフォルト） |
| 位置エンコーディング | ALiBi（学習不要、注意スコアに線形バイアスを加算） |
| 入出力形状 | 入力 `[B, T, 768]` → 出力 `[B, T, 768]` |
| Conformerモジュール順序 | FFN(1/2) → MHSA → Conv → FFN(1/2)（マカロン構造） |
| 勾配フロー | 全パラメータに勾配が伝搬すること |
| パラメータ数 | 約50〜80M（6層Conformerの妥当な範囲） |
| 推論速度 | 10秒音声（500フレーム）に対し100ms以下（A6000基準） |

### 推定作業量
**4〜5日**

### 依存関係
- マイルストーン1完了（HuBERT出力形状の確定）
- `torch>=2.1`
- `einops`（テンソル操作の簡潔な記述）

### 技術的リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| Conformerの正確な仕様が論文に未記載（FFN拡大比、ドロップアウト率、活性化関数など） | 高 | Conformer原論文（Gulati et al., 2020）の標準構成を採用: FFN拡大比4倍、Swish活性化、ドロップアウト0.1。ESPnet/WeNetの実装を参照 |
| ALiBiとConformerの組み合わせが一般的でなく、実装例が少ない | 中 | ALiBi原論文（Press et al., 2022）に基づきヘッドごとの幾何級数スロープ（`m = 2^{-8/n_heads} * 2^{-k}`）を実装。相対位置`(i-j)`にスロープを乗じて注意スコアに加算。正弦波位置エンコーディングやRoPEの代替であり、入力埋め込みに位置情報を加えないことに注意 |
| ストリーミング版との互換性（将来のチャンク因果的注意への拡張） | 中 | 注意マスクを外部から渡せる設計にする。オフライン版ではフルアテンション、ストリーミング版ではチャンク因果マスクを適用できるインターフェースとする |
| Conformerの畳み込みモジュールの因果性（オフラインでは非因果的、ストリーミングでは因果的） | 低 | オフライン版では標準の対称パディングを使用。`causal`フラグを設け、将来の左パディング切り替えに備える |
| ALiBiのヘッド数とスロープの割り当て | 低 | 12ヘッドの場合、スロープは`[2^{-1}, 2^{-1/2}, 2^{-1/4}, ..., 2^{-1/2048}]`ではなく`[2^{-8/12 * k} for k in 1..12]`。原論文のアルゴリズムに厳密に従う |

### ✅ 完了ノート
`stylestream/destylizer/conformer.py`に実装。`alibi.py`にALiBi位置エンコーディングを分離実装。マカロン構造: FFN(0.5)→MHSA(ALiBi)→ConvModule(GLU+DepthwiseConv)→FFN(0.5)→LayerNorm。`causal`フラグでストリーミング拡張対応。34テスト合格。

## マイルストーン3: FSQ（有限スカラー量子化）実装

### タスク名
FSQ [5,3,3]（コードブック45）の実装とSTE勾配伝搬の検証

### 具体的な成果物
- `src/destylizer/fsq.py`: FSQモジュール
  - 下方射影層（線形: 768 → 3）
  - 各次元のスカラー量子化（tanh → スケーリング → 丸め）
  - 上方射影層（線形: 3 → 768）
  - STE（ストレートスルー推定器）による勾配伝搬
- `tests/test_fsq.py`: 単体テスト
  - 量子化出力が正しい離散値のみを取ること
  - STEにより勾配が正しく逆伝搬すること
  - コードブック利用率の計測
  - 情報ボトルネック容量の検証

### 受け入れ基準
| 基準 | 目標値 |
|---|---|
| FSQレベル | [5, 3, 3] |
| 射影次元 D | 3 |
| K値 | K1=2, K2=1, K3=1 |
| 各次元の量子化範囲 | 次元1: {-2,-1,0,1,2}、次元2: {-1,0,1}、次元3: {-1,0,1} |
| コードブックサイズ V | 5 x 3 x 3 = 45 |
| 情報容量 | log2(45) ≒ 5.49 bit/フレーム |
| STE勾配 | `forward`: 量子化値、`backward`: 量子化前の連続値に対する勾配 |
| 入出力形状 | 入力 `[B, T, 768]` → 量子化後 `[B, T, 768]` |
| コンテンツ特徴量 fc の取得 | FSQ下方射影前（Conformer出力）の連続表現を取得可能 |
| コードブック利用率 | 学習収束後に80%以上（45符号中36以上が使用される） |

### 推定作業量
**2〜3日**

### 依存関係
- マイルストーン2完了（Conformer出力を入力として受け取る）

### 技術的リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| FSQの正確な量子化アルゴリズムが不明（tanh + round方式か、直接clamp + round方式か） | 中 | Mentzer et al. (2024) "Finite Scalar Quantization"原論文を参照。推奨実装: `z_hat = round(tanh(z) * K) / K * K` ではなく、`z_q = round(clamp(z, -Ki, Ki))` が一般的。Google Research公式JAX実装をPyTorchに移植する |
| STEの実装ミスで勾配が正しく流れない | 高 | `z_q = z + (round(z) - z).detach()` パターンで実装。これにより順伝搬は離散値、逆伝搬は恒等関数。テストで `loss.backward()` 後にConformer/射影層のパラメータに勾配が存在することを検証 |
| コードブック崩壊（一部の符号のみ使用される） | 中 | 各符号の使用頻度を`wandb`で監視。崩壊が発生する場合: (a) 下方射影の初期化を調整（Xavier均一分布）、(b) FSQ入力のバッチ正規化を検討。ただし論文ではFSQは明示的なコードブック検索がなく崩壊しにくい設計 |
| 推論時のコンテンツ特徴量 fc の取得ポイント | 中 | 「FSQ直前の連続表現」がConformer出力なのか、下方射影後の3次元なのかが曖昧。論文の記述「pre-quantization continuous representation」はConformer出力（768次元）と解釈。根拠: Stylizerの入力次元が768（DiT隠れ層と一致）。3次元ではStylizerへの入力として情報が少なすぎる |
| FSQレベルの選択妥当性の検証 | 低 | アブレーション（表7）を再現する余裕がない場合、まず論文推奨の[5,3,3]で進める。WERが20%を超える場合はレベルを[5,5,3,3]に拡大して切り分け |

### ✅ 完了ノート
`stylestream/destylizer/fsq.py`に実装。STE: `z + (round(clamp(z, -K, K)) - z).detach()`。下方射影(768→3)にXavier均一初期化。`codes_to_indices`/`indices_to_codes`で混合基数エンコーディング。コードブック利用率とperplexity（exp(entropy)）を追跡。26テスト合格。

## マイルストーン4: ASRデコーダ実装

### タスク名
4層トランスフォーマーASRデコーダの実装（学習用、推論時は不使用）

### 具体的な成果物
- `src/destylizer/asr_decoder.py`: ASRデコーダ
  - 自己注意機構（因果マスク付き）
  - 交差注意機構（エンコーダ出力との接続）
  - フィードフォワードネットワーク
  - トークン埋め込み＋出力射影（語彙サイズ分）
- `src/destylizer/tokenizer.py`: テキストトークナイザ（文字レベルまたはサブワード）
- `tests/test_asr_decoder.py`: 単体テスト

### 受け入れ基準
| 基準 | 目標値 |
|---|---|
| デコーダ層数 | 4 |
| 隠れ層次元 | 768 |
| FFN次元 | 3072 |
| 注意ヘッド数 | 12（推定） |
| デコーダタイプ | 自己回帰トランスフォーマー（交差注意によるseq2seq） |
| 入力 | FSQ上方射影後の表現 `[B, T, 768]` |
| 出力 | トークン確率分布 `[B, S, vocab_size]` |
| 損失関数 | 交差エントロピー損失（系列変換ASR損失） |
| トークナイザ | 文字レベル（英語: 約30トークン）またはSentencePiece（論文未記載、CosyVoice 2参照で決定） |
| 注意マスク | デコーダ自己注意に因果マスク適用 |
| 教師強制 | 学習時はground-truthトークンを入力とする教師強制 |

### 推定作業量
**2〜3日**

### 依存関係
- マイルストーン3完了（FSQ上方射影後の出力をデコーダ入力とする）
- テキスト書き起こしの準備（フェーズ1のデータ前処理）

### 技術的リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| ASR損失の種類が不明確（CTC vs 交差注意 vs ハイブリッド） | 高 | 論文は「sequence-to-sequence ASR loss」と明記しており、交差注意デコーダベースのseq2seq損失と解釈するのが妥当。ただし計算コスト削減のため、まずCTC損失（デコーダ不要、Conformer出力に線形層+CTC）で動作確認し、WERが不十分なら交差注意デコーダに切り替える二段階アプローチを採用 |
| トークナイザの選択が論文未記載 | 中 | CosyVoice 2が文字レベル（character-level）ASRを使用しているため、同様に英語文字レベル（a-z, space, apostrophe等、約30トークン）で開始。BPEサブワードはDestylizer学習には不要と判断 |
| ASRデコーダの計算コストが学習を遅延させる | 低 | デコーダは4層と比較的軽量。学習速度が問題になる場合、CTC損失に切り替え（デコーダ不要で高速化）。推論時にはASRデコーダ自体が不要であるため、最終的な速度への影響はない |
| 交差注意デコーダの自己回帰デコーディングが学習を不安定にする | 低 | 教師強制（teacher forcing）で安定的に学習可能。ラベルスムージング（0.1）を適用して過学習を防止 |

### ✅ 完了ノート
`stylestream/destylizer/asr_head.py`に実装。`CTCHead`（線形射影+log_softmax）と`TransformerASRDecoder`（4層、正弦波PE、teacher forcing）の両方を実装。`ASRHead`ファクトリクラスが`loss_type`で切替。CTC: `nn.CTCLoss(blank=0, zero_infinity=True)`。seq2seq: `nn.CrossEntropyLoss(label_smoothing=0.1)`。12テスト合格。

### 技術的リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| 8GPU環境がない場合の学習スケーリング | 高 | 勾配累積（gradient accumulation）で実効バッチサイズを維持。2GPU + 勾配累積4ステップ = 実効バッチ32。学習率のリニアスケーリングルールは適用しない（バッチサイズが同一のため） |
| MSP-Podcastの入手困難（UTDallas研究利用申請が必要） | 中 | 初期実験はLibriTTS（585時間）のみで実施。MSP-Podcastの代わりにESD（Emotional Speech Dataset）を検討。論文表8よりDestylizerは1,300時間で飽和するため、データ量の差は致命的でない |
| 学習の不安定化（損失がNaNや発散） | 中 | (a) 勾配クリッピング（max_norm=1.0）を適用 (b) HuBERT出力のLayerNormを追加 (c) 学習率を1e-5に下げて安定性を確認後、1e-4に引き上げ |
| コサインアニーリングの最小学習率が不明 | 低 | 1e-6を初期値として使用。一般的なASR学習の慣例に従う |
| 可変長音声のバッチ処理効率 | 中 | 動的バッチング（音声長でソートし、近い長さをバッチ化）を実装。パディング比率を最小化し、GPU利用率を向上 |
| 100kステップの学習時間が予算を超過 | 中 | 推定GPU時間: 200〜400時間（8GPU × 25〜50時間）。まず20kステップで損失曲線を確認し、収束傾向を早期判断。mixed precision（fp16/bf16）で学習速度を2倍に高速化 |

### ✅ 完了ノート
`stylestream/destylizer/trainer.py`に`DestylizerTrainer(BaseTrainer)`として実装。`build_model()`でDestylizer構築、`build_dataloader()`でManifest+HuBERT特徴量ロード、`compute_loss()`でCTC/CE損失計算+FSQメトリクス記録。`scripts/train_destylizer.py`をCLIエントリーポイントとして実装（--config, --resume対応）。AdamW β=(0.9, 0.98)、weight_decay=0.01。30テスト合格（モデル統合テスト含む）。

  - バッチ推論、可変長対応

## 判断基準: フェーズ3への移行条件

以下の全条件を満たした場合にフェーズ3（Stylizer実装）に移行する:

| 条件 | 閾値 |
|---|---|
| ASRデコーダWER（LibriSpeech test-clean） | < 15% |
| 話者分類プロービング精度 | < 10% |
| 感情分類プロービング精度 | < 55% |
| アクセント分類プロービング精度 | < 55% |
| FSQコードブック利用率 | > 80% |
| 推論APIが正常動作 | `extract_content_features()` がエラーなく動作 |
| 特徴量にNaN/Infなし | 全テストデータで確認 |

いずれかの条件を満たさない場合、マイルストーン5に戻り以下を検討する:
1. FSQレベルの変更（[5,3,3] → [5,5,3,3] or [3,3,3]）
2. Conformer層数の変更（6 → 4 or 8）
3. 学習ステップの延長（100k → 150k）
4. ASR損失の変更（CTC ↔ seq2seq交差注意）
5. HuBERT抽出層の変更（18 → 16 or 20）
