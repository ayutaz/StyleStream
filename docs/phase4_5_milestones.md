## フェーズ4: ボコーダ（因果的Vocos）

### MS4.1 Vocos解析・因果的畳み込み変換

| 項目 | 内容 |
|---|---|
| **タスク名** | Vocos公式実装の解析と全畳み込み層の因果的変換 |
| **推定作業量** | 3〜5日 |
| **依存関係** | なし（フェーズ2・3と並行可能） |

#### 具体的な成果物

1. **Vocos公式コードの解析レポート**
   - `gemelo-ai/vocos` のConvNextブロック構成の全層リスト
   - 各畳み込み層のカーネルサイズ、ストライド、パディングの一覧表
   - ISTFT復元ヘッドの構造と、因果化対象外の層の特定
   - Vocosの入力メルスペクトログラムの仕様（n_fft、ホップサイズ、メルビン数、周波数範囲）とStyleStream仕様（100ビン、ホップ320、16kHz）との差分

2. **因果的畳み込み変換モジュール `CausalConvNext`**
   - Vocosの全ConvNextブロック内の畳み込み層を因果的畳み込みに変換したモジュール
   - 変換対象: 深さ方向畳み込み（Depthwise Conv1d）のパディングを左側のみに変更
   - 点方向畳み込み（Pointwise Conv1d、カーネルサイズ1）はパディング不要のため変更なし
   - GELU活性化関数、LayerNormは変更なし

3. **因果性検証テスト**
   - 時刻tの出力が時刻t+1以降の入力に依存しないことを数値的に検証するユニットテスト
   - テスト方法: 入力の将来フレームにランダム摂動を加え、過去フレームの出力が不変であることを確認

#### 受け入れ基準

- [x] Vocos公式チェックポイントの全Conv1d層がリストアップされ、因果化対象と非対象が明確に分類されている
- [x] `CausalConvNext` モジュールが、元のVocosと同一の推論結果を（因果パディング変換後の差分を除いて）再現できる
- [x] 因果性テストが全畳み込み層について通過する（将来フレームへの依存ゼロ）
- [x] Vocos公式のメル仕様とStyleStreamの仕様（100ビン、ホップ320、16kHz）の差分が文書化され、変更箇所が特定されている

#### 技術的リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| **Vocosのメル仕様がStyleStreamと不一致** | 高 | Vocos公式デフォルト（n_fft=1024, ホップ=256, 80ビン）からホップ=320, 100ビンへの変更が必要。ISTFTヘッドのn_fft/ホップパラメータを変更し、ウォームスタート時にISTFTヘッドの重みのみランダム初期化する戦略を検討 |
| **ConvNextブロックの因果化で受容野が非対称になる** | 中 | 元のパディング`(kernel_size-1)//2`を`kernel_size-1`（左のみ）に変更。受容野は過去方向に2倍になるが、ストリーミング要件を満たす |
| **Vocos内部のResidual接続やSkip接続が因果性を破る** | 低 | Vocosのアーキテクチャはフレーム間のResidualのみ（時間軸方向の非因果Skipなし）。解析段階で全データフローを検証する |

## フェーズ5: ストリーミング対応

### MS5.1 チャンク因果的注意機構の実装

| 項目 | 内容 |
|---|---|
| **タスク名** | 全コンポーネント共通のチャンク因果的注意機構モジュールの設計・実装 |
| **推定作業量** | 3〜5日 |
| **依存関係** | なし（モジュール単体で実装・テスト可能） |

#### 具体的な成果物

1. **`ChunkedCausalAttention` モジュール**
   - 入力系列を固定チャンクサイズ（デフォルト30フレーム = 600ms @ 50Hz）に分割
   - 各チャンクのクエリは「自チャンク + 全過去チャンク」のキー・バリューにのみ注意
   - 未来チャンクへの注意は注意マスクにより完全にブロック
   - ALiBi位置エンコーディングとの互換性を保証（Destylizer Conformerで使用）
   - チャンクサイズを引数で可変にし、200ms/400ms/600ms/800ms/1000msでの実験に対応

2. **注意マスク生成ユーティリティ**
   - 系列長とチャンクサイズからブロック対角＋下三角の注意マスクを自動生成
   - パディングを含む可変長入力への対応
   - バッチ内で異なる系列長を持つ場合のマスク処理

3. **オフライン注意との等価性テスト**
   - チャンクサイズ = 系列全長に設定した場合、通常のフル注意と完全に同一の結果を返すことを検証
   - チャンク境界でのグラデーション流れの正確性テスト（学習時のバックプロパゲーション）

4. **KVキャッシュ機構**
   - ストリーミング推論用の過去チャンクKVキャッシュ
   - 層ごとにキー・バリューを蓄積し、新チャンク到着時に結合して注意計算
   - メモリ上限管理（リングバッファ5秒 = 250フレーム分を上限）

#### 受け入れ基準

- [x] チャンク因果的注意の出力が、同一チャンク＋過去チャンクのみに依存することが数値テストで検証済み
- [x] チャンクサイズ = 系列長でフル注意と数値的に一致（相対誤差 < 1e-5）
- [x] ALiBi位置エンコーディングとの組み合わせが正しく動作
- [x] KVキャッシュを使った逐次処理と、全系列一括処理で同一結果を返す
- [x] チャンクサイズ30フレーム、系列長500フレームで、フル注意に対して少なくとも30%のメモリ削減を達成

#### 技術的リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| **ALiBiとチャンク境界の相互作用が不明** | 高 | ALiBiの距離計算をチャンク内ローカル位置ではなく、グローバル位置（系列全体での絶対位置）に基づいて行う。オフライン版と同一の位置オフセットを使用 |
| **KVキャッシュのメモリ増大** | 中 | リングバッファサイズ5秒 = 250フレームを上限とし、古いKVを破棄。論文の設計通り、固定長コンテキスト（ターゲット5秒 + ソースリングバッファ5秒 = 最大500フレーム）で運用 |
| **チャンク境界でのアーティファクト** | 中 | チャンクオーバーラップ（数フレーム）を導入し、境界の不連続性を緩和する手法を検討。ただし論文にはオーバーラップの記載がないため、まずオーバーラップなしで実験 |
| **PyTorchの注意マスクの効率的な実装** | 低 | `torch.nn.functional.scaled_dot_product_attention` のFlash Attention 2がブロックスパースマスクを直接サポートしない場合、`xformers` の `BlockDiagonalCausalMask` またはカスタムCUDAカーネルを検討 |

- [x] HuBERTのCNN特徴抽出器が因果的畳み込みに変換され、将来サンプルへの依存がゼロ
- [x] Conformerの畳み込みモジュールが因果化され、将来フレームへの依存がゼロ
- [x] チャンクサイズ=系列長でオフラインDestylizerと数値的に一致
- [x] チャンクサイズ30フレームでの出力が、オフライン版との差分がMS5.4のMSE蒸留で収束可能な範囲にある

#### 技術的リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| **HuBERTの自己注意置換が性能を大幅に劣化させる** | 高 | これが論文でMSE蒸留（MS5.4）が必要とされる主な理由。凍結解除＋ファインチューニングで回復させる。蒸留前の品質劣化度合いをまず定量化し、蒸留の必要ステップ数を見積もる |
| **HuBERTのCNN特徴抽出器の因果化が困難** | 中 | HuBERTのCNN層のカーネルサイズは[10,3,3,3,3,2,2]、ストライドは[5,2,2,2,2,2,2]。各層のパディング変更は機械的に可能だが、受容野の片側化により初期フレームの品質が劣化する可能性がある。初期フレームの出力をマスクまたは無視する戦略を検討 |
| **HuBERTの重みが大きく（約1.2GB）、凍結解除で学習メモリが増大** | 中 | HuBERTの浅い層（例: 1〜12層）は凍結維持し、深い層（13〜24層）のみ凍結解除する部分解凍戦略を検討。メモリが不足する場合は勾配チェックポインティングを適用 |

### MS5.3 リングバッファ・ストリーミング推論パイプライン

| 項目 | 内容 |
|---|---|
| **タスク名** | 固定長ターゲット発話、リングバッファ、チャンク単位推論を統合したストリーミング推論パイプラインの構築 |
| **推定作業量** | 3〜4日 |
| **依存関係** | MS5.1（チャンク因果的注意機構）、MS4.1〜4.2（因果的Vocos） |

#### 具体的な成果物

1. **`StreamingInferencePipeline` クラス**
   - 初期化時にターゲット発話（5秒）を処理し、以下をキャッシュ:
     - ターゲットメルスペクトログラム（250フレーム @ 50Hz）
     - ターゲットのスタイル埋め込みベクトル（768次元、1回だけ抽出）
     - ターゲットのDestylizerコンテンツ特徴量
   - `process_chunk(audio_chunk)` メソッド:
     - 入力: 600ms音声チャンク（9,600サンプル @ 16kHz）
     - 出力: 600ms変換音声チャンク（9,600サンプル @ 16kHz）

2. **リングバッファ管理モジュール**
   - ソースのコンテンツ特徴量を蓄積するリングバッファ（最大5秒 = 250フレーム）
   - FIFO方式: バッファが満杯になったら最古のチャンクを破棄
   - Stylizer入力の構成: [ターゲットメル(250フレーム) | リングバッファ(最大250フレーム)] の結合
   - マスク構成: ターゲット部分は非マスク、ソース部分は100%マスク

3. **チャンク単位CFMサンプリング**
   - 各チャンク（30フレーム）に対してオイラーサンプリング（NFE=16）を実行
   - KVキャッシュを利用した効率的な注意計算
   - CFGスケール（α=2）の適用
   - 出力メル（30フレーム）をボコーダに渡して波形生成

4. **エンドツーエンド推論デモスクリプト**
   - WAVファイル入力（ソース + ターゲット）からチャンク単位で処理し、変換音声を出力
   - 各チャンクの処理時間をプロファイリング・記録
   - リアルタイムファクター（RTF = 処理時間 / チャンク長）の計算

#### 受け入れ基準

- [x] チャンク単位処理の出力を結合した音声が、チャンク境界で不連続性なく滑らかに接続される
- [x] リングバッファが正しくFIFO動作し、5秒を超えた分が破棄されていることのテスト通過
- [x] スタイル埋め込みが初期化時に1回だけ抽出され、各チャンクで再計算されていないことの確認
- [x] 10秒のソース音声に対してエンドツーエンドの推論が完了し、出力音声が生成される
- [x] 各コンポーネント（Destylizer、Stylizer、Vocos）のチャンク処理時間が個別に記録される

#### 技術的リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| **チャンク境界での音声不連続性** | 高 | ボコーダが因果的畳み込みで設計されているため、フレーム単位の出力は原理的に連続。ただし、Stylizerのメル出力がチャンク境界で不連続になる可能性がある。必要に応じてチャンクオーバーラップ（1〜2フレーム）とクロスフェードを導入 |
| **CFMサンプリングのチャンク適用の正当性** | 高 | 各チャンクで独立にノイズ初期化（x0 ~ N(0,I)）してODEを解く。チャンク間でノイズの連続性を保証する仕組みは論文に記載なし。まずは独立サンプリングで実装し、品質問題があればチャンク間でのノイズ補間を検討 |
| **KVキャッシュとリングバッファの組み合わせが複雑** | 中 | ターゲット部分のKVは不変（キャッシュ可能）、ソース部分はリングバッファで更新。注意計算時のインデックス管理を正確に行う。テストケースで境界条件（バッファ空、バッファ満杯、バッファ巻き戻し）を網羅 |
